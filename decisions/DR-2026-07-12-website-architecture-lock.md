## Context

The project has three unreconciled designs for its public-facing website, none of which formally supersedes the others:

1. **`workplan/roadmap-2026-04-27.md` §8** (deprecated 2026-05-08) and **`workplan/website-preparation.md` v2.0** (2026-04-17; its own Phase B tech-stack section is marked superseded by `workplan-co0007-v4.md`, which was itself superseded 2026-05-11): Next.js + PostgreSQL + Directus + Meilisearch + Vercel, 43–70 sessions, 26 parsers, 8 entity types, 20 page types.
2. **`architecture/navigation-modes.md` + `architecture/page-templates.md`** (2026-05-04, status **ACTIVE**, three weeks after the roadmap): a SQLite-only design queried directly against `data/guidebook.db` — a four-door homepage (`/specs`, `/questions`, `/populations`, `/economics`), 14 page templates, spec/question mode toggle. No Next.js/Postgres/Directus/Meilisearch anywhere in it.
3. **`governance/armature_v4.md` + `armature_v4_resolutions.md`** (2026-04-28): a richer *query data model* — 18 functional axes across 6 clusters, plus a new `PopulationCategory` entity — layered on top of, but never integrated into, `page-templates.md`'s flat population-code (`population` table, MOB/VIS/DEAF/…) design.

`roadmap-2026-04-27.md` §9 names the formal gate for this exact reconciliation — **"B7 architecture lock — gates website-preparation Phase B start... Reconcile at B7 lock"** — but B7 has never fired, and every document since has left the conflict standing. Per `audits/project-inventory-and-state-2026-07-12.md` §6.1, this is the actual reason the website hasn't been built: not absence of a plan, but three incompatible plans and no decision closing the loop. That audit, and the owner's request in this session, are the trigger for drafting this DR — but drafting a proposed resolution is not the same act as the owner ratifying one, and this document deliberately keeps that distinction visible rather than blurring it.

Independently, `audits/project-inventory-and-state-2026-07-12.md` §2/§6 established a second, decisive fact: the generator scripts implementing option 1's era (`scripts/generate/population_page.py`, `room_page.py`, `spec_page.py`) do not run against the live database at all — they query tables (`specification`, `population`, `room`) that were never migrated into the canonical schema. Any resolution here must account for this: the committed `site/populations|rooms|specs/` output is a dead prototype, not a partial implementation of any of the three designs above.

## Proposed decision

**Fire B7 architecture lock, closing on option 2, subject to owner ratification:**

1. **Adopt `architecture/navigation-modes.md` + `architecture/page-templates.md`'s SQLite-only design as the sole canonical website architecture.** No Postgres, Directus, Meilisearch, Next.js, or Vercel dependency. The site would be built directly against `data/guidebook.db`, consistent with the project's already-adopted canonical-data-store decision (`governance/repo-strategy-revision-co.md`, D-0138) and with the fact that no such infrastructure exists anywhere in the repo today. *Rationale offered for ratification: standing up four extra services for a solo-author project when the canonical data already lives in SQLite is overhead with no offsetting benefit at this project's scale — but this is a judgment call the owner should confirm, not a fact Claude can establish unilaterally.*
2. **Formally retire the Next.js/PostgreSQL/Directus/Meilisearch/Vercel stack and its 43–70-session estimate** (`roadmap-2026-04-27.md` §8, `website-preparation.md` v2.0), rather than leaving them ambiguously superseded. `website-preparation.md`'s surviving tactical content (glossary, per-file procedures) would remain a historical reference only, per its own existing supersession banner.
3. **Scope `armature_v4`'s 18-axis/`PopulationCategory` model to a deferred v2 enhancement layer, not a v1 blocker.** Per `armature_v4_resolutions.md`'s own "Resolved vs remaining" accounting, every population-specific determination in it (axis-to-population mappings, clinical-scaffold fitness assessments) is **Tier A validation only** — solo-author-against-literature, with Tier B (clinical specialist review) and Tier C (PWLE/DPO review, the CRPD Art 4.3 alignment step) explicitly named as post-launch commitments, not yet done. Building v1 navigation around content the project's own resolutions document says isn't validated yet would repeat the "claims outrunning evidence" pattern the project's audits keep catching elsewhere. `page-templates.md`'s existing flat population-code taxonomy (already used by the live `populations` table) would be used for v1; the axis model would be reconsidered once Tier B/C validation exists to back it.
4. **Do not build `page-templates.md`'s full 14-template schema** (the `specification`, `room`, `conflict`, `jurisdictional_value`, `economics_entry`, `case_study`, `throughline`, `specialist`, `doctrine` tables) as part of this DR. It would remain the target design. A concrete, phased path to it — starting from an honest subset the current schema actually supports — is proposed separately in `workplan/website-v0-path-forward-2026-07-12.md`, so that this DR addresses the *architecture* question without overclaiming that the *content* is ready.

## Consequences if ratified

- `roadmap-2026-04-27.md` §9's B7 gate would close. CS4 (synthesis re-issue cadence) and CS6 (standards monitoring) would **not** be triggered by this closure — both were scoped against Stage B/C machinery from the fully superseded 2026-04-27 roadmap and no longer apply; standards-monitoring automation, if wanted later, should be re-scoped fresh against the current (2026-05-11-derived) workplan stack.
- Future work referencing "the website build" would cite `architecture/navigation-modes.md` + `architecture/page-templates.md` + this DR, not `roadmap-2026-04-27.md` §8 or `website-preparation.md`'s stack section.
- The armature_v4 documents would remain valid as a **data-model design reference** for a v2 axis-based query layer — not retired, only deferred.
- This DR does not resolve the `evidence_cell_state` schema conflict or the Tier-3 `stated`-threshold question (both are addressed in separate, also-PROPOSED, companion DRs dated the same day) and does not itself build any new database table.

## What would make this ACCEPTED

The owner reviewing this document and either (a) changing the Status line and adding a decider note, or (b) directing in a session that these specific calls are approved (with or without amendment). Until then, no other document in this repo should treat items 1–4 above as settled.
