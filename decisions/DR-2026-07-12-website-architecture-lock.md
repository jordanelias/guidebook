## Context

The project has three unreconciled designs for its public-facing website, none of which formally supersedes the others:

1. **`workplan/roadmap-2026-04-27.md` §8** (deprecated 2026-05-08) and **`workplan/website-preparation.md` v2.0** (2026-04-17; its own Phase B tech-stack section is marked superseded by `workplan-co0007-v4.md`, which was itself superseded 2026-05-11): Next.js + PostgreSQL + Directus + Meilisearch + Vercel, 43–70 sessions.
2. **`architecture/navigation-modes.md` + `architecture/page-templates.md`** (2026-05-04, status **ACTIVE**): a four-door homepage (`/specs`, `/questions`, `/populations`, `/economics`), 14 page templates, a spec/question mode toggle.
3. **`governance/armature_v4.md` + `armature_v4_resolutions.md`**: an 18-functional-axis + `PopulationCategory` query data model. `armature_v4.md` self-labels its own status precisely: **"PRE-DECISION DRAFT... NOT decision-quality. Sonnet-drafted; flagged for Opus synthesis at A7/A12"** — it was never in the running as an equal-status design alongside #2's ACTIVE navigation-modes.md; deferring it (below) is less contested than the three-way framing above implies.

`roadmap-2026-04-27.md` §9 names the formal gate for this reconciliation — **"B7 architecture lock... Reconcile at B7 lock"** — but B7 has never fired.

**A critical fact discovered only on adversarial review of this DR's own first draft: option 2 is not actually adoptable as written.** Every one of `page-templates.md`'s 14 "Primary query (SQLite)" blocks is written against tables — `specification`, `population` (singular), `room`, `conflict`, `measurement`, `jurisdictional_value`, `economics_entry`, `case_study`, `throughline`, `specialist`, `doctrine` — that **do not exist in the canonical database** (confirmed directly against `data/guidebook.db`: the live schema has `items`, `populations` — plural, different columns — `bpc_metadata`, `connections`, `gaps`, `evidence_sources`; none of `page-templates.md`'s named tables). This is the *identical* defect already diagnosed in `audits/project-inventory-and-state-2026-07-12.md` for option 1's dead generator scripts (`scripts/generate/population_page.py` queries the same phantom `population` table and fails immediately when run). The first draft of this DR applied that diagnosis to option 1 and failed to apply it to option 2, treating "status ACTIVE" as evidence of adoptability without checking it against the live database. It is not: `page-templates.md`'s *SQL schema* is exactly as aspirational and unbuilt as option 1's. What survives scrutiny is `navigation-modes.md`'s *UX design* (the four-door entry model, the mode-toggle mechanism) — which is real, already partially proven working in `index.html`'s hand-authored mode toggle — separately from `page-templates.md`'s *literal schema*, which is not.

## Proposed decision

**Fire B7 architecture lock on the following, narrower and more precisely scoped terms than "adopt option 2 wholesale":**

1. **Retire Postgres, Directus, and Meilisearch as running services**, and retire Vercel as a hosting dependency. `tools/README.md` documents GitHub Pages (static-only hosting, no server-side execution at any request time) as the project's actual, already-referenced hosting intent. Four extra services for a solo-author project's read-mostly content site are avoidable infrastructure cost. Directus specifically has a sharper reason to go beyond "overhead": `governance/migration-based-writes-adopted-2026-05-11.md` (Status: **Adopted**, unlike the provisional decision the first draft of this DR leaned on — see item 3) commits every `data/guidebook.db` write to flow through versioned, git-mergeable migration files. A CMS admin UI writing live to the database has no migration ledger entry, no atomic commit, no audit trail — structurally incompatible with an already-adopted discipline, not merely redundant with it.
2. **Do not adopt a frontend framework (Next.js or otherwise) as a new dependency — extend the project's own existing, already-working generator pattern instead.** `tools/regenerate_vetting_surface.py` (wired to CI, triggered on every push touching `data/guidebook.db`) already demonstrates the exact pattern a public site needs: query the real live schema, emit a single self-contained static HTML file with data inlined as JSON and vanilla client-side JS for filtering/sorting. `index.html`'s hand-rolled spec/question mode toggle (a `<body>` class flip persisted to `localStorage`) is the same zero-framework approach, already working. This is deliberately a *narrower* claim than "SQLite over Postgres implies no Next.js" (a non sequitur the first draft of this DR made without justification — Next.js's static-export mode can build from a local SQLite file with no runtime server cost at all, so the real argument for skipping it isn't infrastructure cost, it's that the project already has a proven, in-production, zero-new-dependency answer to the same problem). All SQL runs at **static build/generation time only** — never at request time, since GitHub Pages serves static files exclusively; there is no ambiguity here left unresolved, unlike in the first draft.
3. **`/search` gets a concrete answer, not silence.** `navigation-modes.md` names a `/search` route but neither it nor `page-templates.md` specifies how it works, and the first draft of this DR adopted the document "wholesale" without noticing the gap. At this project's actual scale (92 items, 640 evidence sources, 82 `bpc_metadata` rows — all core tables comfortably small), a build-time-generated JSON search index consumed by client-side search (a small JS library, or even a hand-rolled linear filter) requires no backend and no Meilisearch. This is proposed as the concrete resolution, not merely "search is deferred."
4. **`architecture/navigation-modes.md`'s UX design (four-door entry model, mode toggle, page-template *concept*) is adopted as design language for what to build.** `page-templates.md`'s literal SQL schema is **not** adopted — see Context above; adopting it as written would require building 14 tables' worth of unbuilt schema against no proven data model, no better-founded than option 1's dead scripts. The actual page-template/routing/schema design work, scoped against the *real* live tables, is done separately in `workplan/website-v0-path-forward-2026-07-12.md` (written alongside this DR, not merely cited by it — the first draft cited this document before it existed, which is exactly the kind of unenforced claim this whole reconciliation effort exists to eliminate).
5. **`armature_v4`'s 18-axis/`PopulationCategory` model is scoped to a deferred v2 enhancement layer.** Per `armature_v4_resolutions.md`'s own "Resolved vs remaining" accounting, every population-specific determination in it is **Tier A validation only**, with Tier B (clinical specialist) and Tier C (PWLE/DPO, the CRPD Art 4.3 alignment step) explicitly named as post-launch commitments. `page-templates.md`'s flat population-code taxonomy (already used by the live `populations` table) is used for v1.
6. **This DR does not itself build any new database table**, and does not claim `governance/repo-strategy-revision-co.md`'s D-0138 (SQLite-as-datastore) is settled fact — see Revision history: that document's own header reads `Status: PROPOSED — pending project-owner application`, and its anchor decision D-0138 is itself `PROVISIONAL` per `architecture/storage-derivation.md`. This DR's recommendation to use SQLite rests on the same reasoning D-0138 offers (no vendor infra needed at this scale), not on D-0138 being already-ratified — that is a second, still-open owner decision this DR does not resolve on the owner's behalf.

## Consequences if ratified

- `roadmap-2026-04-27.md` §9's B7 gate would close on the terms above — a narrower ratification than "adopt page-templates.md," since that document's literal schema is not being adopted.
- Future work referencing "the website build" would cite `architecture/navigation-modes.md` (UX design only) + this DR + `workplan/website-v0-path-forward-2026-07-12.md` (the actual schema/build plan) — not `page-templates.md`'s literal SQL, and not `roadmap-2026-04-27.md` §8.
- The armature_v4 documents remain valid as a **data-model design reference** for a v2 axis-based query layer — not retired, only deferred, consistent with their own self-declared pre-decision status.
- This DR does not resolve the `evidence_cell_state` schema conflict (a separate, companion DR) and does not itself build any new database table or write any generator code — that is scoped in the path-forward workplan.
