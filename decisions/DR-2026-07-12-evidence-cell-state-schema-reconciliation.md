## Context

Two incompatible schemas exist for `evidence_cell_state`, the empty table meant to hold the project's central per-cell best-practice determination:

**A. `scripts/migrations/024_evidence_cell_state.sql` (already applied; the live, empty table, `user_version=25`):** keyed on `(item_code, population_code)`, FK'd to the real `items` and `populations` tables. Convergence is normalized into a separate, directness-aware `convergence_assessment` table (also empty) carrying `clinical_sources`/`co1_sources`/`co2_sources` and `down_weighted_sources`/`discounted_sources` — ties directly into the Stage 2.2 directness-conditioning work (`schemas/directness.py`) and DR-2026-05-29's scale/directness doctrine. No jurisdiction dimension. No provenance hash.

**B. `workplan/best-practices-assessment-system.md` §4 (2026-06-22; explicitly "sketch; finalize as migration 026," never built):** keyed on `(slug, population, jurisdiction)`. Convergence is a flat `convergence` enum column with no directness distinction. Adds real, valuable capability schema A lacks: `governing_refs` (anti-hallucination), `rule_version` + `derivation_sha` (staleness detection), `tier_basis`, `code_floor_only`, `value_range`, and a `falsification` condition.

## Proposed decision

**Merge, not replace.** Rebuild `evidence_cell_state` (0 rows; safe to rebuild) to:

1. **Keep schema A's cell identity** — `(item_code, population_code)`. Not merely because no canonical `specification`/`slug` table exists (schema A's own original rationale) — a stronger, current reason: `item_bpc_links` (migration 013, built specifically to succeed the single-string `bpc_source_slug` for items governed by more than one BPC, e.g. A-10b) is populated for only **1 of 92 items**. Slug-based identity would today be a strictly *worse* join than item_code, not merely an unavailable one — the intended item↔BPC many-to-many bridge is essentially unbuilt, so choosing slug identity now would inherit that gap directly into the determination engine's Phase 1 backfill.
2. **Do not add a `jurisdiction` dimension to this table at all.** `governance/tier-system.md`'s corridor-width worked example is explicit that "best practice" (evidence-anchored, e.g. 2440mm) and "regulatory floor" (jurisdiction-specific code convergence, e.g. 1800mm) are different *kinds* of claim, and that treating jurisdiction-convergence as best-practice evidence is the named failure mode this doctrine exists to prevent. Baking jurisdiction into `evidence_cell_state`'s identity — even as a nullable/sentinel column — puts a jurisdiction-specific dimension inside a table whose determination is supposed to be jurisdiction-agnostic by definition. Instead:
3. **Add a new, separate `jurisdictional_values` table** normalizing `data/jurisdictional_values/*.yaml` (109 records across 20 files, confirmed not loaded into the DB today) — `item_code`, `jurisdiction`, `standard_name`, `value_numeric`, `unit`, `is_code_minimum`, `evidence_tier` (6 by construction), `source_section`. This is where jurisdiction-specific code floors belong; `evidence_cell_state.code_floor_only` (a boolean flag, per doctrine #2) stays on the determination table without needing a jurisdiction value on the same row, since a Tier-6-only cell can be under-anchored by multiple jurisdictions' codes at once (its `governing_refs` already names the specific Tier-6 sources; their `evidence_sources.jurisdiction` field, or a join to the new table, supplies jurisdiction detail on demand).
4. **Keep schema A's `convergence_assessment` table and FK unchanged** — do not regress to B's flat enum.
5. **Add schema B's provenance/reproducibility columns**: `tier_basis`, `governing_refs` (JSON array, `CHECK (... json_valid(...))`), `rule_version`, `derivation_sha`, `code_floor_only`, `falsification_condition`. Replace B's unstructured `value_range TEXT` with **structured** `value_min REAL, value_max REAL, value_unit TEXT` — matching the numeric/unit convention `jurisdictional_values` (and `source_value_extractions`) already use, so a code-floor-vs-best-practice delta is a plain numeric comparison rather than string parsing.
6. **Do not add a `confidence_bucket` column.** No rule anywhere in the repo derives `high`/`medium`/`low` from `confidence_dimensions_present`/`absent`, and an unenforced, undefined mapping between two representations of the same fact is a driftable redundant field with no consistency check — deferred until a real derivation rule exists to back it, rather than shipped half-designed.
7. **Reuse schema A's existing `created_at`/`created_by_session`/`updated_at`/`updated_by_session` audit columns** instead of B's separate `computed_at`/`computed_by`.
8. **Add `weighting_profile`** exactly as proposed in schema B — no conflict with schema A.
9. **Add the derived views**, with one correction to B's own sketch: `v_best_practice` must exclude `code_floor_only = 1` rows. `best-practices-assessment-system.md` §3's own hard rule states a Tier-6-only cell is `provisional`/`pending`, never the evidence basis for "best practice" — so a view that filters `state IN ('stated','provisional')` without also excluding `code_floor_only=1` would surface exactly the jurisdiction-code-as-best-practice conflation tier-system.md names as the failure mode this whole schema exists to prevent. `v_code_floor_only` is implemented as a join against the new `jurisdictional_values` table (not a bare self-filter), so the code-floor-delta surface `best-practices-assessment-system.md` §4 Phase 4 promises is actually computable from structured columns.
10. **Add validator enforcement now, not as a deferred follow-up**: `scripts/validate_evidence_state.py` is updated (in the same change as this DR) to check (a) `stated`/`provisional` cells have non-empty `governing_refs` — the anti-hallucination gate schema B was designed to provide, which would otherwise exist in name only; (b) `code_floor_only=1` cells cannot be `state='stated'`; (c) the Tier-3-alone threshold from `decisions/DR-2026-07-12-tier3-stated-threshold.md`. All three checks were smoke-tested against synthetic rows (inserted into a throwaway local copy, never the committed DB) confirming they fire on bad data and pass on good data.

This DR does not itself decide the Tier-3 `stated`-threshold policy question — that is `decisions/DR-2026-07-12-tier3-stated-threshold.md`, a real (also proposed, also unratified) evidence-methodology judgment call, though its validator implementation is bundled with this DR's changes since both touch the same function.

## Consequences if ratified

- `scripts/migrations/026_reconcile_evidence_cell_state.sql` and `scripts/migrations/data_20260712150000_jurisdictional-values-backfill.sql` implement the merge and the YAML backfill respectively. Both were dry-run and applied against a local throwaway copy of `data/guidebook.db` (never the committed binary) and verified: schema rebuilds cleanly, all four views execute, the YAML backfill produces exactly 109 rows, and `scripts/validate_evidence_state.py --states-only` passes against the migrated (empty) table.
- `workplan/best-practices-assessment-system.md` §4's SQL sketch is superseded by the actual migration; its phasing plan (§8) is otherwise unaffected.
- The engine/backfill work (§8 Phase 1) is **not done by this DR** — the tables exist with 0 (`evidence_cell_state`) and 109 (`jurisdictional_values`) rows respectively; populating `evidence_cell_state` is separately scoped in `workplan/website-v0-path-forward-2026-07-12.md`.
- `item_bpc_links` being only 1/92 populated is a real, separate forward risk for Phase 1's evidence-gathering step (a slug's evidence needs to reach the right item_code cells) — flagged here as a dependency, not resolved here.

- **v1 (initial draft):** added `jurisdiction` to the table's `UNIQUE` key, added an unspecified `confidence_bucket` column, used unstructured `value_range TEXT`, and did not filter `code_floor_only` out of `v_best_practice`. Cited `D-0138`/`repo-strategy-revision-co.md` as settled in the companion architecture DR (see that DR's own revision history) and referenced this document's Tier-3 companion before it was written.
- **v2 (this version):** independent adversarial review found the jurisdiction-in-key choice reproduces the exact best-practice/code-floor conflation `tier-system.md` names as a failure mode, found `v_best_practice` would have surfaced `code_floor_only` rows as best-practice (a live bug, not a style note), found `confidence_bucket` and `value_range` under-specified, found the anti-hallucination and Tier-3 validator checks were deferred without being scoped as required follow-up, and found the item_code-identity rationale stale (relying on an outdated `bpc_source_slug` gap rather than the current, stronger `item_bpc_links` gap). All are corrected above. `decisions/DR-2026-07-12-tier3-stated-threshold.md` now exists and both its policy and validator implementation are complete.
