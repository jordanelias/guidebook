## Context

Four patterns, of two different kinds: two are genuinely new policy, two are remediation of a rule the architecture doc already states. Bundled in one DR because they share a session and a root theme (git/archive history not being relied on as the version record), but each is independently ratifiable — see "What would make this ACCEPTED."

1. **[REMEDIATION] PI files accumulate instead of archiving on supersession.** `governance/project-instructions-v10_7.md` through `v10_14.md` — 8 files sit side-by-side in `governance/`. This is *not* a versioning-mechanism problem: `project-architecture-guidebook-v2.3.md`'s `<migration_and_growth>` section already prescribes drafting each PI bump as a new `governance/project-instructions-v<next>.md` file, because the PI is not API-writable and the owner needs a complete document to paste into claude.ai project settings, not a diff. That mechanism stays. The actual, narrower gap: v2.3 says how to *create* the next version but not when to *retire* the previous one — so the already-stated `<migration_and_growth>` archive rule ("move to `_archived/` rather than delete") simply isn't being applied to superseded PI files specifically.
2. **[NEW POLICY] Registers snapshotted to dated files instead of queried via git history.** The gap register exists as `_archived/gap_register.md`, `gap_register_archive.md`, and `gap_register_2026-03-20.md`; tier-verification exists as five dated files in `audits/_archived/`, one pair a single day apart. `references/connection-register.md` already demonstrates the alternative: it was replaced with a short redirect stub pointing to the two live files, rather than kept as a third parallel copy.
3. **[REMEDIATION] Six archive-ish locations exist outside the two the architecture already names** (`_archived/`, `workplan/_superseded/`): `misc/archived/`, `_archived/misc-2026/`, `_archived/working-2026/`, `parts/_archived/`, `parts/deprecated/`, `parts/88_to_90/` (89 files total). `architecture/project-architecture-guidebook-v2.3.md` already states content should move to `_archived/` mirroring its origin path — this is a compliance question, not a new rule.
4. **[NEW POLICY] Ten handoff documents with no consistent name or location** (`sessions/handoff-next-session.md`, `sessions/HANDOFF-2026-05-10.md`, `references/phase-b-handoff.md`, three `workplan/*-handoff.md` files, `references/audits/handoff-2026-05-12-*.md`, `working/mobile-app-prototype-v9/HANDOFF.md`, plus two inside a session-artifacts folder).

Note: the armature four-document chain (`armature_v3_review.md`/`v4.md`/`v4_integration.md`/`v4_resolutions.md`) is deliberately **not** included here — per the sweep's correction C, it's a design-review sequence, not version-forking, and belongs in the roadmap's P2 consolidation-into-one-doc item, not this DR's archival-hygiene or new-policy items.

## Proposed decision

1. **[REMEDIATION] PI archival hygiene.** Once a PI version is confirmed pasted-live in claude.ai (or superseded by a newer draft the owner has adopted), the previous version moves to `_archived/governance/` within the same session. The vN-per-revision mechanism itself is unchanged — this only adds the retire step v2.3's general archive rule already implies but doesn't specifically call out for PI.
2. **[NEW POLICY] Register-snapshot ban.** A register (gap register, connection register, decision register, verification-status reports) has exactly one live path. A point-in-time comparison, if genuinely needed, cites a commit SHA or embeds a git-diff excerpt rather than committing a full dated copy.
3. **[REMEDIATION] Single archive location.** All retired content consolidates under `_archived/`, subdirectories mirroring origin path, per the rule `architecture/project-architecture-guidebook-v2.3.md` already states. The six locations named above are folded into it as a follow-up execution pass (not part of this DR).
4. **[NEW POLICY] Handoff convention.** Handoff documents live at `sessions/handoffs/handoff-<topic>-<date>.md`. Existing locations get a short redirect stub (the `connection-register.md` pattern), not silent deletion.

## Consequences if ratified

- **This DR does not execute the consolidation.** Ratification authorizes a follow-up pass to: (a) move PI `v10_7`–`v10_13` to `_archived/governance/`, leaving `v10_14` as the current file at its existing path — no PI content is merged or rewritten, only relocated; (b) delete/redirect the register and verification snapshot files; (c) move the six archive-ish locations under `_archived/`; (d) relocate the 10 handoff documents. Each of these benefits from owner review before execution (e.g. confirming `v10_14` really is the version currently pasted live before archiving the rest), not a mechanical file move Claude should do unilaterally in the same pass as proposing the policy.
- Going forward, new governance-document revisions and new registers follow this policy from the start, preventing the pattern from recurring even before the backlog above is cleared.
- No code, schema, or data changes.
