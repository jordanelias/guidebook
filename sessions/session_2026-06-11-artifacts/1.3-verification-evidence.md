## Verdict

The ledger drift is corrected by a ledger-only backfill plus a one-line
`schema_version` alignment. No entity data changes; the BLOCKING reproducibility
gate stays green. The script is idempotent and safe to re-run.

## What drifted (root cause)

- `scripts/migrate_db.py --dry-run` reported **113 pending** data migrations
  against a DB whose data is in fact already present.
- Cause: migrations applied out-of-band were recorded under ad-hoc slugs the
  runner cannot match (its discovery globs `data_*.sql` and keys on
  `path.stem`), or not recorded at all. The committed `data/guidebook.db`
  already holds every post-cutoff effect — only the `data_migrations` *ledger*
  was wrong.
- Separately, `db_meta.schema_version` read `'11'` while `PRAGMA user_version`
  read `21` (workplan 1.3 R10-d).

## Why ledger-only insert (not a data_*.sql migration, not rebuild-and-replace)

- A `data_*.sql` migration would be re-executed by the runner on every rebuild,
  double-inserting rows. Rejected.
- Rebuild-and-replace is the data-losing trap the DR explicitly rules out:
  job-owned tables in the committed DB exceed what a from-migrations rebuild
  produces (`evidence_source_authors` committed 1244 vs rebuilt 1079;
  `pipeline_runs` committed 10 vs rebuilt 6). Replacing the DB would discard
  165 + 4 rows written by the resolve-DOIs / pipeline jobs. Rejected.
- Correct mechanism: insert the missing ledger rows with `content_sha` computed
  identically to the runner (`sha256` of the file body), so a future rebuild's
  recorded shas line up. The migration SQL is **not** re-executed.

## How the script works

- Reuses the runner's own `discover_data_migrations()` and
  `applied_data_migrations()` so the pending set is provably identical to what
  `migrate_db.py` sees. No reimplementation to drift.
- For each pending `(ts, mid, path)`: `INSERT INTO data_migrations
  (migration_id, applied_at, content_sha, applied_by_session, notes)` with
  `migration_id = path.stem`, `content_sha = sha256(path.read_bytes())`,
  `applied_by_session = reconcile-ledger-DR-2026-05-28`, and a notes string
  recording the out-of-band provenance.
- Aligns `db_meta.schema_version` to `PRAGMA user_version` (upsert).
- Idempotent: the pending filter already excludes applied ids; a second
  per-row guard skips any id already present. Re-running is a no-op.

## DR step-3 — source-verification exemption (RECOMMENDATION, owner-paste)

DR-2026-05-28 step 3 asks how the reproducibility discipline should treat tables
the rebuild cannot reproduce because they are written by external jobs, not
migrations: `evidence_source_authors` and `pipeline_runs`.

**Recommendation: 3a — exempt-by-ownership.** Declare these two tables
job-owned and out of scope for the from-migrations rebuild comparison. The
backfill mechanism is identical under 3a or 3b, so this choice does not affect
Stage 1.3; it governs the *wording* of the architecture/reproducibility
discipline. 3a matches what the CI gate already enforces — the gate compares
seven entity invariants and deliberately omits both tables — so adopting 3a
makes the written doctrine match the gate that is already green, rather than
introducing a comparison the gate does not perform.

**Suggested architecture-doc clause (owner paste):**

> Reproducibility scope. The committed DB is reproducible from migration history
> with respect to the seven gate invariants (schema version; row counts of
> evidence_sources, citation_mining, source_slug_links, gaps, connections,
> items). The tables `evidence_source_authors` and `pipeline_runs` are
> job-owned: they are populated by the resolve-DOIs and pipeline-run workflows,
> not by data migrations, and are exempt from the rebuild-equality check by
> ownership. Their authority is the job log, not the migration ledger.

This is a doctrine-wording change, not executed as part of the backfill. Mint a
doctrine SHA and re-attest per the standing attestation chain if the clause is
adopted.
