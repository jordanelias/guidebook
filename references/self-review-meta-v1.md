# Critical Self-Review: Meta-Analysis Feasibility v1

## Methodological Flaws

### 1. Evidence density ratings are fabricated
The ■/▓/░/· matrix has 99+ cells across populations × categories and populations × rooms. I read ZERO canonical BPC topic-directory files. I read 4 FROZEN flat files (which the project marks as "Any skill reading this file is in error") and inferred ratings from file counts and character lengths. File existence ≠ evidence density. Every cell in §4.1 and §4.2 is ungrounded.

### 2. Evidence type percentages are invented
"~5% Tier 1", "~10% Co-2", "~20% Tier 4" — zero empirical basis. I counted rough "Tier" string matches in 4 frozen files. This violates the project's "I don't know > invention" rule.

### 3. Expected study counts for deep-dives are fabricated
"15-25 poolable studies" for grab bars, "20-40" for acoustics — I have no basis for these numbers. I did no literature search. These are confabulations.

### 4. Conflated meta-analysis with systematic review
The user asked for BOTH. A systematic review doesn't require poolable quantitative data. Qualitative systematic reviews (meta-ethnography, thematic synthesis, framework synthesis) are fully valid for the project's evidence base. By treating "meta-analysable" as the gating criterion, I dismissed ~88% of domains as unsuitable when many are perfectly amenable to systematic review.

### 5. Ignored qualitative meta-synthesis entirely
The project elevates Co-1 (lived experience) to co-primary status. Qualitative meta-synthesis (Noblit & Hare, Thomas & Harden) can pool Co-1 evidence across populations and jurisdictions. This is arguably MORE valuable to the project than quantitative meta-analysis, and I didn't mention it.

### 6. Ignored realist review methodology
The project's evidence base — normative standards + clinical evidence + lived experience + case studies — is textbook material for realist review (Pawson & Tilley). "What works, for whom, in what circumstances" is exactly the project's question. Never mentioned.

### 7. Didn't assess the entire project as requested
The user specified "entire project" as a separate analysis unit. I jumped to component-level decomposition without ever addressing: theoretical coherence, claim-to-evidence ratio, publication readiness, evidence sufficiency for the guidebook's stated purpose.

### 8. Read frozen files instead of canonical sources
The flat BPC files (MOB.md, VIS.md, etc.) are explicitly marked FROZEN. The canonical sources are in the topic directories. I listed 76 topic-directory files by name but read zero.

### 9. Didn't load literature-review-planner skill
This skill exists for exactly this task — PRISMA protocol construction and evidence hierarchy mapping. I produced a review methodology assessment without consulting the project's own methodology tool.

### 10. Co-occurrence assessment ignores existing work
I rated all 12 pairings as having zero evidence. But the project has: cross-population-conflict-mapper outputs, 12 conflict domain analyses, Part 5 content, 170 connection register entries (80 consumed). The conflict-matrices synthesis is substantive evidence for co-occurrence that I completely ignored.

### 11. Economics assessment is ungrounded
I listed 3 BPC economics files but read none. The ratings are pure inference.

### 12. Meta-analysis feasibility treated as binary
It's a spectrum: full Cochrane meta-analysis → mini meta-analysis → vote counting → harvest plot → narrative with quantitative summary → pure narrative. I collapsed this to YES/NO.

### 13. Didn't assess existing Tier 3 systematic reviews
Before recommending new systematic reviews, I should assess whether existing ones (cited in BPC as Tier 3) are adequate, current, or have gaps. No assessment done.

### 14. Didn't use search-logs to assess research completeness
Listed the directory, never checked content.

### 15. "Actionable recommendations" restate existing workplan
"Run the FDR scenarios" and "execute deep-dives" are already queued. The assessment should identify what's NEW.

### 16. Wrong unit of analysis for some domains
Rooms (R-BA, R-KIT) are application contexts, not evidence domains. Evidence clusters around activities and populations, not rooms. Assessing evidence by room is methodologically incorrect — it should be assessed by the functional scenarios that occur in those rooms.

### 17. No consideration of grey literature
Government reports, DPO publications, programme evaluations — these constitute significant evidence for this field but weren't categorised or assessed.

### 18. Connection register evidence not factored
170 connections represent cross-referential synthesis. These are evidence of relationships between specifications, not just administrative entries. Ignoring them understates the project's evidence integration.

## Structural Flaws

### 19. Deliverable doesn't serve a decision
A feasibility assessment should be a decision tool: "invest research resources HERE, not THERE." The v1 deliverable is descriptive, not prescriptive.

### 20. No PRISMA or Cochrane methodology referenced
The deliverable discusses meta-analysis without referencing any established review methodology framework.

## Conclusion

The v1 deliverable violates project rule: "Sources confirmed real. 'I don't know' > invention." The majority of quantitative claims are invented. The review typology is incomplete. The gap map is ungrounded.

**Required for v2:**
1. Read stratified sample of canonical BPC topic-directory files
2. Load literature-review-planner skill
3. Use correct review typology (quantitative MA, qualitative MS, realist review, scoping review)
4. Ground every rating with source evidence or mark [UNVERIFIED]
5. Assess project holistically before decomposing
6. Produce a decision tool, not a descriptive report
